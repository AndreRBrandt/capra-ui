# ADR-019: Design System Contract — Framework-First Visual Ownership

## Status

Aceito

## Contexto

O capra-ui é um framework de componentes para dashboards analíticos. A filosofia de
arquitetura é análoga ao Bootstrap: **o framework define toda a estrutura visual dos
componentes** (espaçamento, tipografia, bordas, cores via tokens, variantes). A app
consumidora **nunca reimplementa nem sobrescreve** o visual dos componentes do framework.

Este princípio nasceu da necessidade de manter consistência visual e evitar o padrão
"CSS local creeping" — onde overrides de estilos crescem nas páginas do app ao longo
do tempo, criando divergências invisíveis e difíceis de manter.

A violação mais sintomática foi: `BaseButton` com todos os estilos em classes Tailwind
que não eram processadas pelo content scan do Tailwind v4 em contexto de library
(`node_modules`), resultando em botões sem cor, sem borda-arredondada e sem padding.
A tentativa de "consertar" isso via `:deep(button)` no app agravou a violação ao invés
de corrigir o problema na raiz.

## Decisão

### Lei de Ownership Visual

```
┌─────────────────────────────────────────────────────────────────┐
│  capra-ui (framework)                                           │
│  ├── Define: estrutura, espaçamento, tipografia, variantes      │
│  ├── Define: tokens padrão em tokens.css                        │
│  └── Componentes: estilos 100% em <style scoped> com var(--*)   │
│                                                                 │
│  capra-analytics (app)                                          │
│  ├── theme.css: SOBRESCREVE tokens de cor (--color-brand-*)     │
│  ├── Props: seleciona variantes pré-definidas (variant="accent")│
│  ├── Slots: injeta CONTEÚDO (não estrutura/estilo)              │
│  └── CSS próprio: APENAS para layout de página e conteúdo       │
│       de slots (loja-detail__grid, etc.) — nunca para           │
│       sobreescrever componentes do framework                    │
└─────────────────────────────────────────────────────────────────┘
```

### O que a app PODE fazer

| Ação | Mecanismo | Exemplo |
|------|-----------|---------|
| Mudar paleta de cores | `theme.css` — override de tokens | `--color-brand-primary: #my-color` |
| Escolher variante visual | Props do componente | `variant="accent"` em BaseButton |
| Injetar conteúdo | Slots de conteúdo | `<template #header>Meu conteúdo</template>` |
| Layout de página próprio | CSS scoped na page | `.loja-detail__grid { display: grid }` |
| Layout de conteúdo em slots | CSS scoped na page | `.minha-tabela-custom { ... }` (dentro do slot body de RecordCard) |

### O que a app NÃO PODE fazer

| Ação proibida | Anti-Pattern | Correto |
|---------------|--------------|---------|
| `:deep(button) { gap: ... }` | AP-16 | Corrigir `gap` no BaseButton do framework |
| `.btn-override { background: red }` | AP-16 | Criar variant no framework ou usar accent |
| `<Button style="border-radius: 0">` | AP-15 + AP-16 | Criar variant `square` no framework |
| `scoped CSS` que sobrescreve framework component | AP-16 | Contribuir ao framework ou usar variant/slot |

### Como implementar componentes no framework

Todo componente do framework (`capra-ui`) que precisa de estilos de cor DEVE usar
`<style scoped>` com CSS custom properties — NUNCA classes Tailwind para cores:

```vue
<!-- ✅ CORRETO — CSS puro no framework, tokens resolvidos pelo app -->
<style scoped>
.base-btn--accent {
  background: var(--color-brand-highlight);
  border-color: var(--color-brand-tertiary);
}
</style>

<!-- ❌ ERRADO — Tailwind não processa node_modules no content scan -->
<template>
  <button class="bg-brand-highlight border-brand-tertiary">
</template>
```

**Por quê:** Tailwind v4 não faz content scan de `node_modules`. Classes utilitárias
como `bg-brand-highlight` nunca são geradas para o bundle da app consumidora.
Scoped CSS com `var(--color-*)` funciona garantidamente porque é CSS nativo resolvido
em runtime pelo browser.

### Como customizar cores na app

O mecanismo correto é sobrescrever tokens em `theme.css` da app:

```css
/* capra-analytics/src/theme.css */

/* Override da paleta da marca para o tema específico do dashboard */
:root {
  --color-brand-primary:    #f5efe8;  /* bege mais quente */
  --color-brand-secondary:  #2d1505;  /* marrom mais escuro */
  --color-brand-highlight:  #d4901a;  /* laranja mais saturado */
}
```

Todos os componentes do framework responderão automaticamente ao override.

## Consequências

### Positivas
- Consistência visual garantida: o framework é dono do visual dos seus componentes
- App impossibilitada de criar "overrides locais que crescem sem controle"
- Theming funciona automaticamente: 1 arquivo CSS (`theme.css`) muda toda a paleta
- Manutenção clara: problema visual num componente → corrigir no framework, nunca no app
- Dark mode: sobrescrever tokens em `[data-theme="dark"]` no `theme.css` da app

### Negativas
- Variantes novas precisam ser adicionadas ao framework (não ad-hoc no app)
- PR ao capra-ui necessário para qualquer mudança visual de componente

## Relação com outras ADRs

- **ADR-010** (Theme System): define o mecanismo `theme.css` + `[data-theme]`
- **ADR-015** (Dependency Boundaries): reforça que app nunca importa do framework para sobrescrever
- **AP-14** (Framework-First): cobre o caso de reimplementar componentes inteiros
- **AP-15** (Design Token Enforcement): cobre hex hardcoded
- **AP-16** (CSS Override): cobre sobrescrever estilos de componentes do framework via CSS
